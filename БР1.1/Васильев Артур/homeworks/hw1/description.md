# Описание проектирования БД — Сайт для поиска работы

## 1. Анализ предметной области

Система обслуживает **два типа пользователей** с разными сценариями:

| Роль | Функционал |
|------|------------|
| **Соискатель (candidate)** | Регистрация, вход, личный кабинет с резюме |
| **Работодатель (employer)** | Регистрация, вход, личный кабинет, управление вакансиями |
| **Общий** | Поиск вакансий с фильтрами, просмотр деталей вакансии |

Ключевые сущности: **пользователь**, **профиль соискателя**, **резюме**, **профиль работодателя**, **вакансия**, **справочники** (отрасль, уровень опыта).

---

## 2. Принципы проектирования

### 2.1. Нормализация (3НФ)

- Данные не дублируются: отрасль и уровень опыта вынесены в **справочники** (`industries`, `experience_levels`), а не хранятся строкой в каждой вакансии.
- Опыт работы и образование — отдельные таблицы, а не JSON в резюме: проще валидировать, индексировать и расширять (например, полнотекстовый поиск по навыкам).

### 2.2. Разделение аутентификации и профилей

Таблица **`users`** хранит только учётные данные (`email`, `password_hash`, `role`). Профили **`candidates`** и **`employers`** — отдельные таблицы со связью 1:1 с `users`.

**Почему так:**

- Один email — одна роль в MVP (соискатель *или* работодатель).
- Нет «пустых» полей компании у соискателя и наоборот.
- При росте системы можно добавить таблицу `user_roles` (M:N), не ломая текущую схему.

### 2.3. Слабая связность через FK

Все связи реализованы **внешними ключами** с каскадным удалением там, где это логично:

| Родитель | Дочерняя таблица | ON DELETE |
|----------|------------------|-----------|
| `users` | `candidates`, `employers` | CASCADE |
| `candidates` | `resumes` | CASCADE |
| `resumes` | `work_experiences`, `educations` | CASCADE |
| `employers` | `vacancies` | CASCADE |

Удаление пользователя-соискателя удаляет профиль, резюме и вложенные записи целиком.

### 2.4. Справочники вместо ENUM в вакансиях

`industries` и `experience_levels` — отдельные таблицы, а не PostgreSQL ENUM.

**Преимущества:**

- Администратор может добавлять отрасли без миграции схемы.
- Фильтрация по `industry_id` и `experience_level_id` — по индексу, быстро и однозначно.
- В API отдаём и `id`, и `name` через JOIN.

### 2.5. Вакансия как центральная бизнес-сущность

`vacancies` агрегирует:

- **Кто** публикует — `employer_id`
- **Категория** — `industry_id`
- **Требуемый опыт** — `experience_level_id`
- **Контент** — `title`, `description`, `requirements`
- **Фильтры** — `salary_from`, `salary_to`, `salary_currency`, `location`
- **Жизненный цикл** — `is_published` (черновик / опубликована)

Поле `is_published` позволяет работодателю сохранять черновик без показа в поиске.

### 2.6. Резюме 1:1 с соискателем

В MVP — **одно резюме** на `candidate_id` (UNIQUE на `candidate_id`).

Для учебного проекта этого достаточно. Расширение до нескольких резюме: убрать UNIQUE, добавить `is_primary`.

## 3. Типы данных и ограничения

| Решение | Обоснование |
|---------|-------------|
| `uuid` для PK | Безопасность (не угадываемые id), удобство при распределённых системах |
| `email` UNIQUE | Один аккаунт на email |
| `user_id` UNIQUE в `candidates` / `employers` | Строго один профиль на пользователя |
| `salary_from` / `salary_to` nullable | Вакансия может быть «по договорённости» |
| `end_date` nullable в `work_experiences` | Текущее место работы |
| `sort_order` | Упорядочивание блоков резюме на клиенте |

---

## 4. Соответствие функционалу сайта

| Требование | Таблицы / поля |
|------------|----------------|
| Вход / регистрация | `users` |
| ЛК соискателя с резюме | `candidates`, `resumes`, `work_experiences`, `educations` |
| Поиск с фильтрами | `vacancies` + JOIN `industries`, `experience_levels`; фильтр по зарплате и `is_published = true` |
| Детали вакансии | `vacancies` + JOIN `employers` (компания) |
| ЛК работодателя | `employers`, CRUD `vacancies` |

---

## 5. Возможные расширения (вне MVP)

- `applications` — отклики соискателя на вакансию (M:N `candidates` ↔ `vacancies`)
- `favorite_vacancies` — избранное
- `refresh_tokens` — отдельная таблица для OAuth2-подобного refresh
- Полнотекстовый поиск: `tsvector` на `vacancies.title`, `description`

Текущая схема намеренно компактна, но **нормализована** и готова к этим расширениям без переписывания ядра.
