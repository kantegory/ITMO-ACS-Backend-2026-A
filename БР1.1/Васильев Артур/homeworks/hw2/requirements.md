# Требования к бэкенд-приложению — Job Search API

## 1. Выбор формата API

**Выбран формат: REST API**

| Критерий | REST | BFF |
|----------|------|-----|
| Независимость клиентов | Web, mobile, admin — один контракт | Отдельный BFF под каждый фронт |
| Кэширование | HTTP-кэш, CDN для GET | Сложнее |
| Документация | OpenAPI / Swagger — стандарт | Зависит от реализации |
| Сложность бэкенда | Умеренная | Выше (слой агрегации) |

Для учебного проекта и темы «сайт поиска работы» REST оптимален: ресурсо-ориентированные URL (`/vacancies`, `/resumes`), предсказуемые методы HTTP, единая спецификация OpenAPI.

**Версионирование:** префикс `/api/v1`.

**Формат данных:** JSON, кодировка UTF-8.

**Аутентификация:** JWT Bearer (`Authorization: Bearer <token>`).

## 2. Функциональные требования

### 2.1. Аутентификация и авторизация

| ID | Требование | Роль |
|----|------------|------|
| FR-AUTH-01 | Регистрация соискателя (email, password, full_name) | — |
| FR-AUTH-02 | Регистрация работодателя (email, password, company_name) | — |
| FR-AUTH-03 | Вход по email и password, выдача JWT | — |
| FR-AUTH-04 | Доступ к защищённым эндпоинтам только с валидным JWT | — |
| FR-AUTH-05 | Разграничение: соискатель не управляет чужими вакансиями; работодатель — только своими | candidate / employer |

### 2.2. Соискатель — личный кабинет и резюме

| ID | Требование |
|----|------------|
| FR-CAND-01 | Получение своего профиля |
| FR-CAND-02 | Обновление профиля (ФИО, телефон, город) |
| FR-CAND-03 | Получение резюме (включая опыт работы и образование) |
| FR-CAND-04 | Создание / обновление резюме (title, summary, skills) |
| FR-CAND-05 | CRUD записей опыта работы в резюме |
| FR-CAND-06 | CRUD записей образования в резюме |

### 2.3. Работодатель — личный кабинет и вакансии

| ID | Требование |
|----|------------|
| FR-EMP-01 | Получение профиля компании |
| FR-EMP-02 | Обновление профиля компании |
| FR-EMP-03 | Список своих вакансий (включая черновики) |
| FR-EMP-04 | Создание вакансии |
| FR-EMP-05 | Редактирование своей вакансии |
| FR-EMP-06 | Удаление своей вакансии |
| FR-EMP-07 | Публикация / снятие с публикации (`is_published`) |

### 2.4. Вакансии (публичная часть)

| ID | Требование |
|----|------------|
| FR-VAC-01 | Поиск вакансий с пагинацией |
| FR-VAC-02 | Фильтр по отрасли (`industry_id`) |
| FR-VAC-03 | Фильтр по зарплате (`salary_min` — вакансии, где `salary_to >= salary_min` или вилка пересекается) |
| FR-VAC-04 | Фильтр по уровню опыта (`experience_level_id`) |
| FR-VAC-05 | Текстовый поиск по названию (опционально `q`) |
| FR-VAC-06 | Детальная страница вакансии: описание, требования, компания, отрасль, опыт, зарплата |
| FR-VAC-07 | В поиске только опубликованные (`is_published = true`) |

### 2.5. Справочники

| ID | Требование |
|----|------------|
| FR-REF-01 | Список отраслей |
| FR-REF-02 | Список уровней опыта |

## 3. Нефункциональные требования

### 3.1. Производительность

| ID | Требование |
|----|------------|
| NFR-PERF-01 | Время ответа GET `/vacancies` (p95) ≤ 500 мс при до 10 000 вакансий |
| NFR-PERF-02 | Пагинация по умолчанию: `limit=20`, максимум `limit=100` |

### 3.2. Надёжность и доступность

| ID | Требование |
|----|------------|
| NFR-REL-01 | Транзакционность при создании пользователя + профиля |
| NFR-REL-02 | Graceful shutdown HTTP-сервера |
| NFR-REL-03 | Health-check `GET /health` для оркестратора (Docker) |

### 3.3. Безопасность

| ID | Требование |
|----|------------|
| NFR-SEC-01 | Пароли хранятся как bcrypt-хеш, не в открытом виде |
| NFR-SEC-02 | JWT с ограниченным TTL (например, 24 ч) |
| NFR-SEC-03 | Валидация входных данных на границе API |
| NFR-SEC-04 | Защита от IDOR: работодатель меняет только свои `vacancy_id` |
| NFR-SEC-05 | CORS настраивается через переменные окружения |

### 3.4. Масштабируемость и сопровождаемость

| ID | Требование |
|----|------------|
| NFR-MAIN-01 | Слоистая архитектура: handler → usecase → repository |
| NFR-MAIN-02 | Слабая связность через интерфейсы (контракты) |
| NFR-MAIN-03 | Миграции БД версионируются (golang-migrate) |
| NFR-MAIN-04 | OpenAPI-спецификация актуальна и лежит в `docs/` |

### 3.5. Наблюдаемость

| ID | Требование |
|----|------------|
| NFR-OBS-01 | Структурированное логирование запросов (method, path, status, duration) |
| NFR-OBS-02 | Единый формат ошибок в JSON |

### 3.6. Развёртывание

| ID | Требование |
|----|------------|
| NFR-DEP-01 | Контейнеризация приложения и PostgreSQL (Docker Compose) |
| NFR-DEP-02 | Конфигурация через переменные окружения |

## 4. Модель ошибок API

Все ошибки возвращаются в теле:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Human readable message",
    "details": []
  }
}
```

| HTTP | code | Когда |
|------|------|-------|
| 400 | `VALIDATION_ERROR` | Невалидное тело запроса, query |
| 401 | `UNAUTHORIZED` | Нет / невалидный JWT |
| 403 | `FORBIDDEN` | JWT есть, но роль / владелец не подходят |
| 404 | `NOT_FOUND` | Ресурс не найден |
| 409 | `CONFLICT` | Email уже занят, резюме уже существует |
| 500 | `INTERNAL_ERROR` | Непредвиденная ошибка сервера |

## 5. Соответствие БД (hw1) и эндпоинтам

| Ресурс API | Таблицы БД |
|------------|------------|
| User / Auth | `users` |
| Candidate profile | `candidates` |
| Resume | `resumes`, `work_experiences`, `educations` |
| Employer profile | `employers` |
| Vacancy | `vacancies` + JOIN |
| References | `industries`, `experience_levels` |
