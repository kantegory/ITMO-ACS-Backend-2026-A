## 2. Анализ исходного монолита

### 2.1. Функциональные блоки монолита

До рефакторинга приложение Edulytica представляло собой монолитное FastAPI-приложение, включавшее:

| Компонент | Описание |
|-----------|----------|
| **Authentication** | Регистрация, логин, JWT-токены (access + refresh), email-верификация |
| **Account management** | Управление профилем пользователя (имя, фамилия, организация) |
| **Document management** | Загрузка, парсинг, хранение документов |
| **RAG pipeline** | Retrieval-Augmented Generation — обогащение промптов контекстом из ChromaDB |
| **Orchestration** | Координация многозадачных AI-пайплайнов с очередями сообщений |
| **LLM inference** | Вызов Qwen и Vikhr моделей |
| **Events/feedback** | Управление конференциями (events), custom events, фидбек |
| **Tickets** | Управление тикетами (заявками на проверку) |
| **Backup** | Автоматическое резервное копирование БД |

### 2.2. Проблемы монолита

- **Масштабирование**: разные компоненты имеют разные профили нагрузки (LLM требует GPU, API — CPU)
- **Изоляция сбоев**: ошибка в RAG-пайплайне могла заблокировать весь API
- **Независимая разработка**: сложно разделить работу между командами
- **Релизный цикл**: изменение в одном модуле требовало перезапуска всего приложения

## 4. Протоколы взаимодействия

### 4.1. Синхронное взаимодействие (HTTP/REST)

- **Client → Gateway**: внешние запросы через Nginx
- **Gateway → Services**: проксирование по HTTP с перезаписью URL (rewrite rules)
- **Orchestrator → API**: финализация отчёта через HTTP POST (внутренний endpoint `/api/internal/v1/upload_report`)
- **Orchestrator → RAG**: обогащение промптов через HTTP-клиент `RagClient`

### 4.2. Асинхронное взаимодействие (Kafka)

- **Orchestrator → Kafka → LLM**: задача отправляется в топик `llm_tasks.qwen` или `llm_tasks.vikhr`
- **LLM → Kafka → Orchestrator**: результат возвращается через `llm_tasks.result`

Kafka-топики:
| Топик | Партиции | Реплики | Назначение |
|-------|----------|---------|-----------|
| `llm_tasks.result` | 1 | 1 | Результаты LLM |
| `llm_tasks.qwen` | 1 | 1 | Задачи для Qwen |
| `llm_tasks.vikhr` | 1 | 1 | Задачи для Vikhr |

## 5. Декомпозиция базы данных

Единая PostgreSQL БД используется всеми сервисами (фаза 1). В перспективе — разделение на отдельные схемы/инстансы.

**Основные таблицы:**
- `accounts` / `users` — пользователи и профили
- `events` / `custom_events` — конференции (5 типов)
- `files` — загруженные документы
- `tickets` — заявки на проверку
- `feedback` — отзывы
- `tickets_history` / `ticket_finished_data` — история проверок

---

## 6. Безопасность

- **JWT-аутентификация**: Access + Refresh токены с настраиваемым временем жизни
- **Email-верификация**: код подтверждения по email (SMTP)
- **Internal API Secret**: секретный заголовок `X-Internal-Secret` для межсервисных вызовов
- **CORS**: настроен для localhost и разрешённых origins

---

## 7. Мониторинг и health checks

Каждый микросервис имеет endpoint `/health` для проверки работоспособности. Docker Compose использует healthcheck'и с интервалами от 5 до 60 секунд в зависимости от сервиса.

---

## 8. Выводы

Предложенная микросервисная архитектура позволяет:
1. **Независимо масштабировать** компоненты (GPU-сервисы отдельно от CPU)
2. **Изолировать сбои** — отказ одного сервиса не блокирует остальные
3. **Параллельно разрабатывать** сервисы разными командами
4. **Использовать разные технологии** под разные задачи (FastAPI, Nginx, Kafka)

Архитектура соответствует DDD-принципам: каждый сервис владеет своей предметной областью и имеет чёткие границы ответственности.
