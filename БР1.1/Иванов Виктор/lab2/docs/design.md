# Технический дизайн микросервисной архитектуры (ДЗ4)

Документ описывает разбиение монолита из `lab1` (REST API бронирования столиков) на микросервисы с принципом **database-per-service**, синхронные внутренние HTTP API и асинхронные события через **RabbitMQ**.

## 1. Границы сервисов

| Сервис | Ответственность | База данных |
|--------|-----------------|-------------|
| **user-service** | регистрация, вход, JWT; внутренний просмотр профиля | `user.sqlite` |
| **restaurant-service** | рестораны, столики, отзывы (ссылка на пользователя по `user_id`); журнал входящих событий бронирования | `restaurant.sqlite` |
| **booking-service** | жизненный цикл бронирований, проверка конфликтов по времени | `booking.sqlite` |
| **notification-service** | фоновая обработка событий (лог уведомлений) | `notification.sqlite` |

Между сервисами **нет общих таблиц** и **нет общего файла SQLite**: только идентификаторы (`user_id`, `restaurant_id`, `table_id`) как внешние ссылки.

## 2. Способы взаимодействия

1. **Синхронно (HTTP)**  
   - Клиент обращается к `user-service` и `booking-service` с JWT.  
   - `booking-service` перед созданием брони вызывает:  
     - `GET /internal/users/{id}` — убедиться, что пользователь существует;  
     - `GET /internal/tables/{tableId}?restaurant_id=` — проверить столик и вместимость.  
   - Заголовок `X-Internal-Token` защищает внутренние маршруты (см. `openapi-inter-service.yaml`).

2. **Асинхронно (RabbitMQ)**  
   - После успешного сохранения брони `booking-service` публикует JSON в **fanout**-exchange `booking_events`.  
   - Подписчики: `restaurant-service` (очередь `booking_events_restaurant`, запись в `booking_event_log`) и `notification-service` (очередь `booking_events_notifications`, запись в `notification_log`).  
   - Паттерн аналогичен цепочке «корзина → оплата → склад» из [примера mentoring](https://github.com/kantegory/mentoring/tree/master/26_express_rabbitmq), но с **fanout**, чтобы одно событие дошло до нескольких сервисов.

## 3. Разделение данных (database-per-service)

- **Пользователи** — только в БД user-service (пароль в виде хэша).  
- **Рестораны, столики, отзывы** — только в БД restaurant-service. Отзыв хранит `user_id` без FK на другую БД.  
- **Бронирования** — только в БД booking-service; связи с пользователем и рестораном логические (UUID).  
- **Логи уведомлений** — только в БД notification-service.

## 4. Шаги миграции с монолита

1. Выделить сущности и миграции по сервисам; отключить перекрёстные FK между БД.  
2. Реализовать внутренние эндпоинты и единый `INTERNAL_SERVICE_TOKEN`.  
3. Перенести сценарий создания брони: валидация пользователя и столика через HTTP; конфликт слотов — локально в booking-service.  
4. Поднять RabbitMQ (docker compose), ввести публикацию события `booking.created` и подписчиков.  
5. Обновить клиентов (Postman/фронт): базовые URL и порты разных сервисов.  
6. Наблюдаемость: логи консюмеров, при необходимости DLX для очередей.

## 5. Ошибки межсервисных вызовов

| Код | Ситуация |
|-----|----------|
| 403 | неверный `X-Internal-Token` |
| 404 | пользователь или столик не найден |
| 502 | user-service / restaurant-service недоступен (ошибка в `booking-service`) |

Асинхронная доставка: при сбое обработки сообщение `nack` без requeue (в учебной версии); в проде — DLX и повторные попытки.
