## Задача

Реализовать разделение backend-приложения на микросервисы.

## Ход работы

В рамках работы монолитное API из ЛР1 было разделено по доменным областям:

- `auth-service` - регистрация, авторизация и проверка токена;
- `catalog-service` - справочники, компании и вакансии;
- `applicant-service` - резюме и отклики соискателя;
- `employer-service` - управление вакансиями и статусами откликов;
- `api-gateway` - единая точка входа для клиента.

Публичные запросы идут через API Gateway по адресу `/api/v1`. Внутреннее взаимодействие между сервисами выполняется по HTTP:

- applicant-service проверяет пользователя через auth-service;
- applicant-service проверяет вакансию через catalog-service;
- employer-service проверяет работодателя через auth-service;
- employer-service меняет статус отклика через applicant-service;
- employer-service управляет вакансиями через catalog-service.

Для упрощения проверки все сервисы запускаются одной командой, но слушают разные порты.

## Проверка

Основной сценарий покрыт тестом:

```bash
go test ./...
```

Сценарий проверки:

1. вход соискателя;
2. получение списка вакансий;
3. создание резюме;
4. отклик на вакансию;
5. вход работодателя;
6. изменение статуса отклика на `reviewing`.

## Вывод

В результате работы backend-приложение было разделено на несколько HTTP-микросервисов с отдельными зонами ответственности и единым API Gateway.
