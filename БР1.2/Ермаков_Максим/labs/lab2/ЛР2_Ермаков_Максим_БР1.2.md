# Лабораторная работа №2

## Реализация микросервисов

**Студент:** Ермаков Максим Олегович  
**Группа:** K3340  
**Тема:** разделение backend-приложения бронирования ресторанов на микросервисы

## Цель работы

Цель лабораторной работы — реализовать разделение монолитного REST API из ЛР1
на набор микросервисов, опираясь на технический дизайн, подготовленный в ДЗ4.
Разделение выполнено с соблюдением подхода `database-per-service`: каждый
доменный сервис владеет собственной базой данных и не обращается напрямую к БД
других сервисов.

## Состав реализации

В рамках ЛР2 реализованы следующие runtime-компоненты:

| Компонент | Порт | Назначение |
|---|---:|---|
| `api-gateway` | `8100` | Публичная точка входа `/api/v1`, проверка JWT, маршрутизация и композиция ответов |
| `identity-service` | `8101` | Пользователи, регистрация, вход, профиль, проверка JWT |
| `catalog-service` | `8102` | Рестораны, локации, кухни, фотографии и публикация ресторанов |
| `menu-service` | `8103` | Категории меню и позиции меню |
| `reservation-service` | `8104` | Столики, доступность столиков и бронирования |
| `review-service` | `8105` | Отзывы и пересчёт рейтинга ресторанов |

## Разделение баз данных

Для микросервисов предусмотрены отдельные PostgreSQL-БД:

| Сервис | База данных | Основные таблицы |
|---|---|---|
| `identity-service` | `identity_db` | `users` |
| `catalog-service` | `catalog_db` | `locations`, `cuisines`, `restaurants`, `restaurant_cuisines`, `restaurant_photos` |
| `menu-service` | `menu_db` | `menu_categories`, `menu_items` |
| `reservation-service` | `reservation_db` | `restaurant_tables`, `reservations` |
| `review-service` | `review_db` | `reviews` |

Связи между БД не реализуются внешними ключами. Например, `reservation-service`
хранит `user_id` и `restaurant_id`, но проверяет пользователя через
`identity-service`, а ресторан — через `catalog-service`.

## Авторизация и права доступа

Пользовательский клиент работает только с `api-gateway`. JWT-токен передаётся
в заголовке:

```http
Authorization: Bearer <accessToken>
```

Gateway вызывает внутренний endpoint `identity-service`:

```http
POST /internal/auth/introspect
```

После успешной проверки gateway передаёт во внутренние сервисы доверенный
контекст пользователя:

```http
X-User-Id: <uuid>
X-User-Role: USER|ADMIN
X-Request-Id: <uuid>
```

Внутренние сервисы используют эти заголовки для проверки владельца ресурса и
административных прав. Например, `reservation-service` разрешает пользователю
работать только со своими бронированиями, а административные операции требуют
роль `ADMIN`.

## Межсервисное взаимодействие

Реализованы синхронные REST-вызовы между сервисами:

- `api-gateway -> identity-service` для регистрации, входа, профиля и
  introspection JWT;
- `api-gateway -> catalog/menu/reservation/review` для маршрутизации публичного
  API;
- `reservation-service -> catalog-service` для проверки существования и
  публикации ресторана;
- `reservation-service -> identity-service` и `catalog-service` для обогащения
  ответа по бронированию;
- `review-service -> catalog-service` для проверки ресторана и обновления
  денормализованного рейтинга;
- `review-service -> identity-service` для обогащения отзывов данными автора.

## Сохранение публичного API

Публичный контракт `/api/v1` сохранён на стороне `api-gateway`. Клиентские
сценарии из ДЗ3 могут продолжать обращаться к тем же основным endpoint'ам:

- `/api/v1/auth/register`;
- `/api/v1/auth/login`;
- `/api/v1/users/me`;
- `/api/v1/reference/*`;
- `/api/v1/restaurants/*`;
- `/api/v1/reservations/*`;
- `/api/v1/admin/*`.

## Запуск

Подробная инструкция запуска находится в `README.md` папки ЛР2.

Кратко:

```bash
npm install
docker compose up -d
npm run build
npm run start:identity
npm run start:catalog
npm run start:menu
npm run start:reservation
npm run start:review
npm run start:gateway
```

Каждый `start:*` запускается в отдельном терминале.

## Вывод

В результате лабораторной работы монолитное backend-приложение было разделено
на набор микросервисов по границам, спроектированным в ДЗ4. Каждый доменный
сервис получил собственную БД, публичный API был сохранён через gateway, а
межсервисное взаимодействие реализовано через внутренние REST-вызовы. Такое
разделение подготавливает проект к следующим работам: подключению брокера
сообщений, контейнеризации каждого сервиса и настройке CI/CD.
