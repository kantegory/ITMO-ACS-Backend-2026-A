## Цель работы

Цель работы — подключить брокер сообщений RabbitMQ и реализовать межсервисное
взаимодействие через очередь сообщений. Работа выполнена в папке `labs/lab3`,
которая является отдельной копией `labs/lab2`, чтобы версия ЛР2 оставалась
самостоятельно запускаемой.

## Выбор брокера

Для реализации выбран RabbitMQ, потому что для текущей архитектуры достаточно
надежной topic-очереди, подтверждения обработки сообщений и удобного
management-интерфейса для демонстрации на защите.

В `docker-compose.yml` добавлен контейнер:

```yaml
rabbitmq:
  image: rabbitmq:3-management
  ports:
    - 5672:5672
    - 15672:15672
```

Параметры подключения:

| Параметр | Значение |
|---|---|
| AMQP URL | `amqp://maindb:maindb@127.0.0.1:5672` |
| Management UI | `http://127.0.0.1:15672` |
| Логин / пароль | `maindb` / `maindb` |
| Exchange | `restaurant.events` |
| Тип exchange | `topic` |

PostgreSQL для `lab3` опубликован на порту `25432`, а сервисы используют порты
`8200-8205`, чтобы новая версия не конфликтовала с `lab2`, где используются
`15432` и `8100-8105`.

## Реализованный сценарий

В ЛР2 рейтинг ресторана после создания или изменения отзыва обновлялся
синхронным HTTP-вызовом из `review-service` в `catalog-service`.

В ДЗ5 этот сценарий переведен на асинхронное взаимодействие:

1. Пользователь создает или изменяет отзыв.
2. `review-service` сохраняет отзыв в `review_db`.
3. `review-service` пересчитывает средний рейтинг и количество отзывов.
4. `review-service` публикует событие `review.rating.recalculated` в RabbitMQ.
5. `catalog-service` читает событие из очереди `catalog.restaurant-rating`.
6. `catalog-service` обновляет поля `avgRating` и `reviewsCount` в `catalog_db`.

## Событие

Routing key:

```text
review.rating.recalculated
```

Payload:

```json
{
  "eventId": "uuid-or-request-id",
  "occurredAt": "2026-05-26T17:00:00.000Z",
  "source": "review-service",
  "restaurantId": "uuid",
  "avgRating": 5,
  "reviewsCount": 1
}
```

## Изменения в коде

| Файл | Назначение |
|---|---|
| `src/common/message-bus.ts` | Общее подключение к RabbitMQ, публикация и потребление событий |
| `src/common/events.ts` | Типы и routing key событий |
| `src/services/review/app.ts` | Публикация события после пересчета рейтинга |
| `src/services/catalog/app.ts` | Consumer очереди и применение нового рейтинга ресторана |
| `docker-compose.yml` | PostgreSQL + RabbitMQ |
| `.env` | Переменные подключения к RabbitMQ |
| `scripts/smoke-test.mjs` | Проверка, что рейтинг обновился через очередь |

## Проверка

```bash
npm install
docker compose up -d
npm run build
npm run start:identity
npm run start:catalog
npm run start:menu
npm run start:reservation
npm run start:review
npm run start:gateway
npm run smoke
```

В smoke-тесте после создания отзыва выполняется повторное чтение карточки
ресторана через gateway. Если `avgRating` и `reviewsCount` обновились, значит
событие прошло через RabbitMQ и было обработано `catalog-service`.

## Вывод

В результате работы к микросервисной архитектуре подключен RabbitMQ, а один из
межсервисных сценариев переведен с синхронного HTTP-вызова на асинхронную
обработку события. Это уменьшает связность `review-service` и `catalog-service`
и подготавливает проект к дальнейшему развитию событийной архитектуры.
