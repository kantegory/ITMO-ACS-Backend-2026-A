# ДЗ4: технический дизайн микросервисной архитектуры

В этой папке подготовлены материалы для домашней работы №4 по разделению
монолитного backend-приложения бронирования ресторанов на микросервисы.

## Файлы

- `microservices-design.md` — основной проектный документ: границы сервисов,
  базы данных, схемы взаимодействия, сценарии и план миграции.
- `internal-openapi.yaml` — OpenAPI 3.1-спецификация внутренних REST-эндпоинтов
  для межсервисного взаимодействия.
- `Отчёт_ДЗ4_Ермаков_К3340.docx` — отчёт для сдачи, собранный из проектного
  документа.
- `build_report.py` — вспомогательный скрипт сборки DOCX из Markdown-исходника.
- `server.mjs` — локальный сервер Swagger UI/ReDoc для защиты.

## Запуск Swagger

Из папки `homeworks/hw4`:

```bash
npm start
```

После запуска документация будет доступна по адресам:

- `http://127.0.0.1:8004/docs`
- `http://127.0.0.1:8004/docs/redoc`
- `http://127.0.0.1:8004/docs/openapi.yaml`

Можно поменять порт:

```bash
PORT=8010 npm start
```

Сервер не требует установки npm-зависимостей: он использует встроенный HTTP
модуль Node.js. Swagger UI и ReDoc загружаются с CDN.

## Краткая идея разделения

Монолит предлагается разделить на шесть runtime-компонентов:

1. `api-gateway` — единая публичная точка входа, маршрутизация и композиция
   ответов.
2. `identity-service` — пользователи, роли, регистрация, вход и проверка JWT.
3. `catalog-service` — рестораны, локации, кухни, фотографии и публикация.
4. `menu-service` — категории меню и позиции меню.
5. `reservation-service` — столики, доступность и бронирования.
6. `review-service` — отзывы и расчёт рейтинга ресторанов.

Каждый сервис, кроме `api-gateway`, владеет собственной базой данных. Прямые
внешние ключи между БД заменяются на идентификаторы внешних агрегатов и
внутренние REST-вызовы.
