# Требования к backend-приложению

## 1. Назначение системы

Система предназначена для поиска ресторанов и бронирования столиков. Основные пользователи:

- `USER` — регистрируется, авторизуется, ищет рестораны, просматривает карточку ресторана, оставляет отзывы и управляет своими бронированиями.
- `ADMIN` — управляет справочными и контентными сущностями, публикует рестораны и изменяет статусы бронирований.

## 2. Функциональные требования

### 2.1 Аутентификация и учетная запись

- Пользователь может зарегистрироваться по имени, фамилии, email, телефону и паролю.
- Пользователь может войти в систему по email и паролю.
- Пользователь может выйти из системы.
- Пользователь может просматривать и редактировать свой профиль.
- У каждого пользователя есть роль: `ADMIN` или `USER`.

### 2.2 Каталог ресторанов

- Пользователь может получить список ресторанов.
- Пользователь может фильтровать рестораны по кухне, городу, району, станции метро и ценовой категории.
- Пользователь может открывать детальную страницу ресторана.
- В карточке ресторана должны отображаться:
  - общая информация;
  - локация;
  - кухни;
  - меню;
  - фотографии;
  - средний рейтинг;
  - отзывы.

### 2.3 Бронирование столиков

- Пользователь может посмотреть доступные столики ресторана на конкретную дату, время и количество гостей.
- Пользователь может создать бронирование.
- Пользователь может изменить своё бронирование.
- Пользователь может отменить своё бронирование.
- Пользователь может получить историю своих бронирований.
- Бронирование имеет статусы: `PENDING`, `CONFIRMED`, `CANCELLED`, `COMPLETED`.

### 2.4 Отзывы

- Пользователь может получить список отзывов ресторана.
- Авторизованный пользователь может оставить отзыв.
- Авторизованный пользователь может изменить свой отзыв.
- Рейтинг ресторана должен рассчитываться на основе отзывов.

### 2.5 Администрирование

- Администратор может создавать и редактировать локации.
- Администратор может создавать и редактировать рестораны.
- Администратор может публиковать и скрывать рестораны.
- Администратор может управлять столиками ресторана.
- Администратор может управлять категориями меню и позициями меню.
- Администратор может загружать и изменять фотографии ресторана.
- Администратор может изменять статус бронирования.

### 2.6 Справочники

- Система должна предоставлять справочники кухонь, локаций, ролей, ценовых категорий и статусов бронирований.

## 3. Нефункциональные требования

### 3.1 Архитектурные

- API реализуется в формате `REST API`.
- Обмен данными осуществляется в формате `JSON`.
- Для описания контракта используется `OpenAPI 3.1`.
- Документация должна быть доступна через `Swagger UI` и `ReDoc`.

### 3.2 Безопасность

- Закрытые методы требуют заголовок `Authorization: Bearer <token>`.
- Пользователь может работать только со своими бронированиями и своими отзывами.
- Административные методы доступны только роли `ADMIN`.
- Пароли хранятся в зашифрованном виде и не возвращаются клиенту.

### 3.3 Качество данных

- Все входные данные проходят валидацию.
- Для бизнес-конфликтов используются ошибки `409 Conflict`.
- Для ошибок валидации используется `422 Validation Error`.
- Для отсутствующих сущностей используется `404 Not Found`.

### 3.4 Производительность и удобство использования

- Списочные методы поддерживают пагинацию.
- Поиск ресторанов должен поддерживать комбинирование нескольких фильтров.
- Ответы API должны быть предсказуемыми и единообразными по структуре.

### 3.5 Надежность и сопровождение

- Ошибки должны возвращаться в едином формате с кодом и сообщением.
- Все ключевые действия должны логироваться на backend-стороне.
- API должно быть расширяемым без ломающих изменений для уже опубликованных клиентов.

## 4. Принятые допущения

- Все системные временные метки возвращаются строками в формате ISO 8601.
- Для бронирований используются поля:
  - `reservationDate` в формате `YYYY-MM-DD`;
  - `reservationTime` в формате `HH:mm`.
- Ценовые категории в схеме API заданы как `LOW`, `MEDIUM`, `HIGH`, так как конкретные значения enum в ДЗ1 не были зафиксированы.
