# ДЗ2: Требования к backend-приложению

## 1) Функциональные требования

1. Пользователь может зарегистрироваться по `name`, `email`, `password`.
2. Пользователь может войти по `email`, `password` и получить JWT токен.
3. Авторизованный пользователь может просмотреть свой личный кабинет (`профиль`).
4. Пользователь может получить список ресторанов.
5. Список ресторанов поддерживает фильтрацию по:
   - кухне (`cuisine`)
   - городу (`city`)
   - диапазону средней цены (`priceMin`, `priceMax`)
6. Пользователь может открыть страницу ресторана с:
   - основной информацией
   - меню
   - фото
   - отзывами
7. Авторизованный пользователь может создать бронирование в ресторане.
8. Пользователь может просмотреть историю своих бронирований.
9. API возвращает корректные HTTP-коды и стандартизированные ошибки.

## 2) Нефункциональные требования

1. Формат API: REST API (JSON over HTTP).
2. Время ответа базовых запросов (листинги/детали) в пределах ~500ms на локальном окружении.
3. Безопасность:
   - пароли хранятся только в хешированном виде (`bcrypt`)
   - приватные эндпоинты защищены JWT
4. Масштабируемость по слоям:
   - разделение на `models`, `controllers`, `routes`, `middleware`
5. Поддерживаемость:
   - единый формат ошибок
   - OpenAPI-схема для документации и проверки контракта
6. Переносимость:
   - запуск на Node.js через `npm`
   - хранилище SQLite для учебной локальной демонстрации

## 3) Выбранный формат API

Выбран **REST API**:
- проще для учебного проекта и демонстрации в Postman;
- прозрачно маппится на CRUD/ресурсную модель (`users`, `restaurants`, `reservations`);
- хорошо документируется через OpenAPI 3.0.
