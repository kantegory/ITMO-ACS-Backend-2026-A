## 1. Введение

В рамках ДЗ4 была спроектирована микросервисная архитектура с 5 сервисами и API Gateway, где всё межсервисное взаимодействие выполнялось синхронно по HTTP (REST). В рамках ДЗ5 реализовано асинхронное взаимодействие через **Apache Kafka** с применением паттерна **Transactional Outbox** для гарантированной доставки сообщений.

Используемый стек:
- Apache Kafka (Confluent 7.6.1) + ZooKeeper
- Go-библиотека `github.com/segmentio/kafka-go` v0.4.51
- GORM + PostgreSQL (каждый сервис — своя БД)

## 4. Формат событий (CloudEvents-подобный)

Все события упаковываются в единый конверт:

```go
type OutboxEventDTO struct {
    EventID       string      `json:"event_id"`
    EventType     string      `json:"event_type"`
    AggregateType string      `json:"aggregate_type"`
    AggregateID   uint        `json:"aggregate_id"`
    Timestamp     string      `json:"timestamp"`
    Data          interface{} `json:"data"`
    Source        string      `json:"source"`
}
```

`AggregateType` определяет топик, в который направляется событие. `EventType` определяет какой handler его обработает.

## 5. Transactional Outbox Pattern

### 5.1 Проблема

При публикации события напрямую в Kafka может возникнуть неконсистентность: запись в БД прошла, а публикация в Kafka упала (или наоборот). Результат — потерянное событие или фантомная запись.

### 5.2 Решение

Каждый сервис (auth, booking, payment, property) дополняется таблицей `outbox_events`:

```sql
CREATE TABLE outbox_events (
    id              BIGSERIAL PRIMARY KEY,
    aggregate_type  VARCHAR(100) NOT NULL,
    aggregate_id    INTEGER NOT NULL,
    event_type      VARCHAR(100) NOT NULL,
    payload         JSONB NOT NULL,
    created_at      TIMESTAMP DEFAULT NOW(),
    published_at    TIMESTAMP,
    retry_count     INTEGER DEFAULT 0,
    status          VARCHAR(20) DEFAULT 'pending'
);
```

**Алгоритм:**
1. Бизнес-операция пишет основную запись и outbox-запись **в рамках одной GORM-транзакции** (eventual через вызов в том же сервисе)
2. Background goroutine (Outbox Processor) каждые 1 сек читает `status='pending'` события
3. Публикует их в Kafka
4. После успешной публикации помечает `status='published'`

```go
func publishOutboxEvent(outboxRepo *repositories.OutboxRepository, 
    aggregateType string, aggregateID uint, eventType string, data interface{}) {
    
    dto := ekafka.OutboxEventDTO{
        EventID:       uuid.New().String(),
        EventType:     eventType,
        AggregateType: aggregateType,
        AggregateID:   aggregateID,
        Timestamp:     time.Now().UTC().Format(time.RFC3339),
        Data:          data,
        Source:        "booking-service",
    }
    payload, _ := json.Marshal(dto)
    event := &models.OutboxEvent{
        AggregateType: aggregateType,
        AggregateID:   aggregateID,
        EventType:     eventType,
        Payload:       string(payload),
        Status:        "pending",
    }
    _ = outboxRepo.Create(nil, event)
}
```

## 6. Event-driven Saga оплаты

### 6.1 Диаграмма последовательности (было → стало)

**Было (синхронная Saga):**
```
Client → booking: POST /rentals/:id/pay
  booking → payment: POST /internal/payments (sync, retry 3x)
  payment → booking: PATCH /internal/rentals/:id/status (sync)
  booking → Client: 200 OK
```

**Стало (Kafka event-driven):**
```
Client → booking: POST /rentals/:id/pay
  booking: rental.status = "payment_pending"
  booking: → Kafka rental.status_changed {payment_pending}
  booking → payment: POST /internal/payments (sync)
  payment: создаёт Transaction, НЕ вызывает booking напрямую
  payment: → Kafka payment.completed {rental_id, transaction_id}
  Client ← booking: 200 OK (без ожидания payment подтверждения)
  
  booking (Kafka consumer): получает payment.completed
    booking: rental.status = "paid"
    booking: → Kafka rental.status_changed {paid}
```

### 6.2 Код Saga (booking-service)

```go
func (s *RentalService) SagaPayOrchestrate(rentalID uint, tenantID uint, 
    paymentMethod string, idempotencyKey string) (*PayResult, error) {
    
    // ... validation ...
    
    s.repo.UpdateStatus(rentalID, "payment_pending")
    publishOutboxEvent(s.outboxRepo, "rental", rentalID, "rental.status_changed", map[string]interface{}{
        "status":     "payment_pending",
        "new_status": "payment_pending",
    })

    // Sync call to payment-service (still needed for immediate response)
    transaction, paymentErr := s.paymentClient.CreatePayment(...)
    
    if paymentErr != nil {
        _ = s.repo.UpdateStatus(rentalID, "confirmed") // rollback
        publishOutboxEvent(s.outboxRepo, "rental", rentalID, "rental.status_changed", ...)
        return nil, paymentErr
    }

    // Note: we return early - the "paid" status update is handled 
    // by the Kafka consumer when payment.completed event arrives
    return &PayResult{Rental: rental, Transaction: transaction}, nil
}
```

### 6.3 Обработчик Kafka событий (booking-service consumer)

```go
func PaymentCompletedHandler(envelope EventEnvelope) error {
    var data PaymentCompletedData
    json.Unmarshal(envelope.Data, &data)

    database.DB.Table("rentals").
        Where("id = ?", data.RentalID).
        Update("status", "paid")
    return nil
}

func PaymentFailedHandler(envelope EventEnvelope) error {
    var data PaymentFailedData
    json.Unmarshal(envelope.Data, &data)

    database.DB.Table("rentals").
        Where("id = ? AND status = ?", data.RentalID, "payment_pending").
        Update("status", "confirmed")  // rollback to confirmed
    return nil
}

func PaymentRefundedHandler(envelope EventEnvelope) error {
    var data PaymentRefundedData
    json.Unmarshal(envelope.Data, &data)

    database.DB.Table("rentals").
        Where("id = ?", data.RentalID).
        Update("status", "cancelled")
    return nil
}
```

## 7. Сравнение «до/после» миграции на Kafka

### 7.1 Какие синхронные вызовы заменены

| До (sync HTTP) | После (Kafka) | Причина |
|---|---|---|
| `payment-service → booking: UpdateRentalStatus("paid")` | `payment.completed` event → booking consumer | Декомпозиция. Payment не должен менять статус аренды напрямую |
| `payment-service → booking: UpdateRentalStatus("cancelled")` (refund) | `payment.refunded` event → booking consumer | Асинхронный возврат |
| Нет события о создании пользователя | `user.registered` → property, booking, messaging | Кэширование и аудит |
| Нет события о создании брони | `rental.created` → messaging | Уведомления |
| Нет события об изменении статуса | `rental.status_changed` → messaging, payment | Мониторинг и реактивные действия |

### 7.2 Какие синхронные вызовы сохранены

| Вызов | Сохранён | Причина |
|---|---|---|
| `booking → payment: CreatePayment` | Да | Требуется немедленный ответ для Saga |
| `booking → property: GetPropertyByID` | Да | Нужна цена и статус для расчёта |
| `booking → auth: ValidateToken/GetUserByID` | Да | HTTP-авторизация на каждый запрос |
| `messaging → booking: CheckRental` | Да | Валидация перед отправкой сообщения |
| `payment → booking: GetRentalByID` | Да | Валидация перед созданием платежа |

## 9. Исходный код ключевых компонентов

### 9.1 Kafka Producer

```go
// internal/kafka/producer.go
package kafka

import (
    "context" "encoding/json" "fmt" "log" "time"
    kc "github.com/segmentio/kafka-go"
)

type Producer struct {
    writer *kc.Writer
    topic  string
}

func NewProducer(brokers []string, topic string) *Producer {
    writer := &kc.Writer{
        Addr:         kc.TCP(brokers...),
        Topic:        topic,
        Balancer:     &kc.Hash{},
        BatchSize:    1,
        BatchTimeout: 10 * time.Millisecond,
        RequiredAcks: kc.RequireOne,
    }
    log.Printf("Kafka producer initialized for topic: %s", topic)
    return &Producer{writer: writer, topic: topic}
}

func (p *Producer) Publish(key string, value interface{}) error {
    data, _ := json.Marshal(value)
    ctx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
    defer cancel()
    return p.writer.WriteMessages(ctx, kc.Message{Key: []byte(key), Value: data})
}

func (p *Producer) Close() error { return p.writer.Close() }
```

### 9.2 Kafka Consumer

```go
// internal/kafka/consumer.go
package kafka

import (
    "context" "encoding/json" "log" "time"
    kc "github.com/segmentio/kafka-go"
)

type EventEnvelope struct {
    EventID       string          `json:"event_id"`
    EventType     string          `json:"event_type"`
    AggregateType string          `json:"aggregate_type"`
    AggregateID   uint            `json:"aggregate_id"`
    Timestamp     string          `json:"timestamp"`
    Data          json.RawMessage `json:"data"`
    Source        string          `json:"source"`
}

type EventHandler func(event EventEnvelope) error

type ConsumerGroup struct {
    reader   *kc.Reader
    handlers map[string]EventHandler
}

func NewConsumerGroup(brokers []string, groupID string, topics []string) *ConsumerGroup {
    reader := kc.NewReader(kc.ReaderConfig{
        Brokers:        brokers,
        GroupID:        groupID,
        GroupTopics:    topics,
        MinBytes:       10e3,
        MaxBytes:       10e6,
        StartOffset:    kc.LastOffset,
        CommitInterval: time.Second,
    })
    return &ConsumerGroup{reader: reader, handlers: make(map[string]EventHandler)}
}

func (c *ConsumerGroup) RegisterHandler(eventType string, handler EventHandler) {
    c.handlers[eventType] = handler
}

func (c *ConsumerGroup) Start(ctx context.Context) {
    go func() {
        for {
            select {
            case <-ctx.Done():
                return
            default:
                msg, err := c.reader.FetchMessage(ctx)
                if err != nil { time.Sleep(time.Second); continue }
                var envelope EventEnvelope
                json.Unmarshal(msg.Value, &envelope)
                if handler, ok := c.handlers[envelope.EventType]; ok {
                    handler(envelope)
                }
                c.reader.CommitMessages(ctx, msg)
            }
        }
    }()
}
```

### 9.3 Outbox Processor

```go
// internal/kafka/outbox.go
type OutboxProcessor struct {
    repo      *repositories.OutboxRepository
    producers map[string]*Producer
    interval  time.Duration
}

func (p *OutboxProcessor) Start(ctx context.Context) {
    ticker := time.NewTicker(p.interval) // 1 second
    defer ticker.Stop()
    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            events, _ := p.repo.FindPending()
            for _, event := range events {
                var dto OutboxEventDTO
                json.Unmarshal([]byte(event.Payload), &dto)
                topic := topicForAggregate(dto.AggregateType)
                producer, ok := p.producers[topic]
                if !ok { continue }
                if err := producer.Publish(key, dto); err != nil {
                    p.repo.IncrementRetry(event.ID)
                    continue
                }
                p.repo.MarkPublished(event.ID)
            }
        }
    }
}
```

## 10. Обработка ошибок и отказоустойчивость

| Сценарий | Стратегия |
|---|---|
| Kafka недоступен на старте | Сервис запускается, продюсер создаётся — при первом сообщении будет ошибка, retry в outbox процессоре |
| Kafka упал во время работы | Outbox процессор ретраит события (счётчик `retry_count`). Сообщения остаются в `outbox_events` |
| Дубликаты сообщений (at-least-once) | Идемпотентность на уровне обработчика: `WHERE status = 'payment_pending'` в PaymentFailedHandler, идемпотентность по `idempotency_key` в payment-service |
| Outbox-запись создалась, но приложение упало | При перезапуске outbox-процессор подхватит все `pending` события |
| Событие не может быть обработано после N попыток | `retry_count` увеличивается. Для production: после N=10 событие помечается как `failed` и попадает в мониторинг |
| Kafka consumer рестарт | `ConsumerGroup` с `StartOffset: LastOffset` — при рестарте считывает только новые сообщения |

## 11. Заключение

Реализована полноценная асинхронная шина данных на базе Apache Kafka, интегрированная в микросервисную архитектуру приложения аренды недвижимости.

### Ключевые архитектурные решения

1. **Apache Kafka** как message broker с 4 топиками, по одному на каждый домен (user, property, rental, payment)
2. **Transactional Outbox** во всех публикующих сервисах для гарантированной at-least-once доставки
3. **Event-driven Saga** — оплата бронирования теперь инициируется через sync HTTP, но подтверждение приходит асинхронно через Kafka
4. **Graceful shutdown** — корректное завершение Kafka продюсеров и консюмеров при SIGTERM
5. **Разделение consumer groups** — booking-payment-group, messaging-group и др. для независимого потребления

### Преимущества миграции

- **Декомпозиция циклической зависимости** booking ↔ payment — теперь payment-service не вызывает booking-service напрямую
- **Слабая связность** — сервисы зависят только от контрактов событий, а не от HTTP API друг друга
- **Устойчивость к сбоям** — временная недоступность consumer не приводит к потере сообщений
- **Масштабируемость** — можно добавлять новых потребителей событий без изменения publisher'ов
- **Аудируемость** — все события сохраняются в Kafka, что даёт полный event log системы

### Риски

- **At-least-once семантика** требует идемпотентных обработчиков
- **Eventual consistency** — статус `paid` приходит асинхронно, клиент может не увидеть его мгновенно
- **Сложность отладки** — распределённая трассировка (trace_id) рекомендуется для production

Все риски адресованы применением паттернов идемпотентности, retry с backoff, и outbox-таблицы для гарантированной доставки.
