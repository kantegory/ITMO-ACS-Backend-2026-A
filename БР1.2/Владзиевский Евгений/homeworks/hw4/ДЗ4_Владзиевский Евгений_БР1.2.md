## 1. Анализ текущего монолита

Текущее приложение — монолитный backend на Go (Gin + GORM + PostgreSQL), реализующий REST API для аренды недвижимости.

### Текущая ER-диаграмма

```
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────────┐
│  property_types │     │    amenities     │     │  property_amenities │

│ id (PK)         │     │ id (PK)          │     │ property_id (PK,FK) │
│ name            │     │ name             │     │ amenity_id (PK,FK)  │
└────────┬────────┘     │ icon             │     └──────────┬──────────┘
         │               │ description      │                │
         │               └──────────────────┘                │
         │                                                   │
         ▼                                                   ▼
┌─────────────────────────────────────────────────────────────────────┐
│                            properties                                │
│─────────────────────────────────────────────────────────────────────│
│ id (PK) │ owner_id (FK→users) │ type_id (FK→property_types)        │
│ title │ description │ price_per_day │ city │ address                │
│ latitude │ longitude │ status │ created_at                          │
```

### Ключевые проблемы монолита

| Проблема | Описание |
|----------|----------|
| Единая точка отказа | Падение одного компонента останавливает всю систему |
| Сложность масштабирования | Нельзя масштабировать только узкое место (например, поиск свойств) |
| Сильная связность | Сервисы напрямую зависят от репозиториев друг друга |
| Единая БД | Все 8 таблиц в одной PostgreSQL, общая схема — невозможно независимое развёртывание |
| Деплой целиком | Любое изменение требует деплоя всего приложения |

## 2. Ограниченные контексты и декомпозиция

На основе предметно-ориентированного проектирования (DDD) выделены 5 ограниченных контекстов (bounded contexts), каждый из которых становится отдельным микросервисом.

### 2.1. Auth Service (Сервис аутентификации и пользователей)

| Аспект | Описание |
|--------|----------|
| **Ответственность** | Регистрация, аутентификация, управление профилем, ролевая модель |
| **Данные** | `users` |
| **Бизнес-правила** | Хеширование паролей, JWT-токены (access + refresh), валидация ролей |
| **Публичное API** | `/api/v1/auth/register`, `/api/v1/auth/login`, `/api/v1/users/me`, `/api/v1/users/:id/role` |
| **Межсервисное API** | Валидация токенов, получение данных пользователя по ID, проверка роли |

### 2.2. Property Catalog Service (Сервис каталога недвижимости)

| Аспект | Описание |
|--------|----------|
| **Ответственность** | CRUD объектов недвижимости, типы недвижимости, удобства, изображения |
| **Данные** | `property_types`, `amenities`, `property_amenities`, `properties`, `property_images` |
| **Бизнес-правила** | Владелец или админ управляет своими объектами; изображения сохраняются на диск; фильтрация и поиск |
| **Публичное API** | `/api/v1/properties`, `/api/v1/property-types`, `/api/v1/amenities`, `/api/v1/properties/:id/images` |
| **Межсервисное API** | Получение свойства по ID, проверка владения, проверка доступности на даты |

### 2.3. Booking Service (Сервис бронирования)

| Аспект | Описание |
|--------|----------|
| **Ответственность** | Создание, просмотр, изменение статуса бронирований (аренды) |
| **Данные** | `rentals` |
| **Бизнес-правила** | Валидация дат (будущее, start < end), отсутствие пересечений, ролевая авторизация, конечный автомат статусов |
| **Публичное API** | `/api/v1/rentals` |
| **Межсервисное API** | Проверка участия пользователя в аренде, получение аренд по свойству/пользователю |

### 2.4. Payment Service (Сервис платежей и транзакций)

| Аспект | Описание |
|--------|----------|
| **Ответственность** | Обработка платежей, ведение истории транзакций, идемпотентность |
| **Данные** | `transactions` |
| **Бизнес-правила** | Идемпотентность по ключу, Saga-оркестрация при оплате, запись всех операций |
| **Публичное API** | `/api/v1/transactions` |
| **Межсервисное API** | Создание платежа (принимает команду от Booking Service), запрос статуса транзакции |

### 2.5. Messaging Service (Сервис сообщений)

| Аспект | Описание |
|--------|----------|
| **Ответственность** | Отправка и получение сообщений между арендаторами и владельцами |
| **Данные** | `messages` |
| **Бизнес-правила** | Только участники аренды могут переписываться, сообщение 1-2000 символов, группировка по диалогам |
| **Публичное API** | `/api/v1/messages` |
| **Межсервисное API** | Проверка участия в диалоге по свойству |

## 3. Разделение базы данных (database-per-service)

Каждый микросервис владеет собственной базой данных. Прямые JOIN-запросы между сервисами заменяются на вызовы по API или репликацию событий.

### 3.1. Схема БД Auth Service

```sql
CREATE DATABASE auth_db;

CREATE TABLE users (
    id          SERIAL PRIMARY KEY,
    email       VARCHAR(150) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    full_name   VARCHAR(100) NOT NULL,
    phone       VARCHAR(20),
    role        VARCHAR(20) CHECK (role IN ('tenant','owner','admin')) DEFAULT 'tenant',
    created_at  TIMESTAMP DEFAULT NOW()
);
```

### 3.2. Схема БД Property Catalog Service

```sql
CREATE DATABASE property_catalog_db;

CREATE TABLE property_types (
    id   SERIAL PRIMARY KEY,
    name VARCHAR(50) UNIQUE NOT NULL
);

CREATE TABLE amenities (
    id          SERIAL PRIMARY KEY,
    name        VARCHAR(100) UNIQUE NOT NULL,
    icon        VARCHAR(50),
    description TEXT
);

CREATE TABLE properties (
    id            SERIAL PRIMARY KEY,
    owner_id      INTEGER NOT NULL,  -- FK логический (данные в Auth Service)
    type_id       INTEGER REFERENCES property_types(id) ON DELETE SET NULL,
    title         VARCHAR(255) NOT NULL,
    description   TEXT,
    price_per_day DECIMAL(12,2) CHECK (price_per_day > 0) NOT NULL,
    city          VARCHAR(100) NOT NULL,
    address       TEXT NOT NULL,
    latitude      DECIMAL(9,6),
    longitude     DECIMAL(9,6),
    status        VARCHAR(20) CHECK (status IN ('active','archived')) DEFAULT 'active',
    created_at    TIMESTAMP DEFAULT NOW()
);

CREATE TABLE property_amenities (
    property_id INTEGER REFERENCES properties(id) ON DELETE CASCADE,
    amenity_id  INTEGER REFERENCES amenities(id) ON DELETE CASCADE,
    PRIMARY KEY (property_id, amenity_id)
);

CREATE TABLE property_images (
    id          SERIAL PRIMARY KEY,
    property_id INTEGER REFERENCES properties(id) ON DELETE CASCADE,
    image_url   TEXT NOT NULL,
    is_main     BOOLEAN DEFAULT FALSE
);
```

**Важное изменение:** `owner_id` в таблице `properties` больше не является внешним ключом на уровне БД — теперь это логическая ссылка. Проверка существования пользователя выполняется через API Auth Service.

### 3.3. Схема БД Booking Service

```sql
CREATE DATABASE booking_db;

CREATE TABLE rentals (
    id          SERIAL PRIMARY KEY,
    tenant_id   INTEGER NOT NULL,        -- логический FK → auth_db.users
    property_id INTEGER NOT NULL,        -- логический FK → property_catalog_db.properties
    start_date  DATE NOT NULL,
    end_date    DATE NOT NULL,
    total_price DECIMAL(12,2) NOT NULL,
    status      VARCHAR(20) CHECK (status IN ('pending','confirmed','paid','cancelled','finished'))
                DEFAULT 'pending',
    created_at  TIMESTAMP DEFAULT NOW(),
    CHECK (end_date > start_date)
);

CREATE INDEX idx_rentals_property_dates ON rentals(property_id, start_date, end_date);
CREATE INDEX idx_rentals_tenant ON rentals(tenant_id);
```

### 3.4. Схема БД Payment Service

```sql
CREATE DATABASE payment_db;

CREATE TABLE transactions (
    id              SERIAL PRIMARY KEY,
    rental_id       INTEGER NOT NULL,         -- логический FK → booking_db.rentals
    tenant_id       INTEGER NOT NULL,         -- денормализован для независимых запросов
    amount          DECIMAL(12,2) NOT NULL,
    payment_method  VARCHAR(50),
    idempotency_key VARCHAR(64) UNIQUE,
    type            VARCHAR(20) CHECK (type IN ('payment','refund')) DEFAULT 'payment',
    status          VARCHAR(20) CHECK (status IN ('success','failed','pending','refunded'))
                    DEFAULT 'success',
    created_at      TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_transactions_rental ON transactions(rental_id);
CREATE INDEX idx_transactions_tenant ON transactions(tenant_id);
```

**Обоснование денормализации `tenant_id`:** платёжный сервис должен уметь формировать историю транзакций пользователя без вызова Booking Service (decoupling). Данные записываются один раз при создании транзакции и не меняются.

### 3.5. Схема БД Messaging Service

```sql
CREATE DATABASE messaging_db;

CREATE TABLE messages (
    id           SERIAL PRIMARY KEY,
    sender_id    INTEGER NOT NULL,      -- логический FK → auth_db.users
    receiver_id  INTEGER NOT NULL,      -- логический FK → auth_db.users
    property_id  INTEGER NOT NULL,      -- логический FK → property_catalog_db.properties
    message_text TEXT NOT NULL,
    sent_at      TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_messages_conversation ON messages(property_id, sender_id, receiver_id);
```

### 3.6. Сводная таблица разделения БД

| Микросервис | База данных | Таблицы | Зависимости от других БД |
|-------------|-------------|---------|--------------------------|
| Auth Service | `auth_db` | `users` | Нет |
| Property Catalog | `property_catalog_db` | `property_types`, `amenities`, `property_amenities`, `properties`, `property_images` | `owner_id` → `auth_db.users` (логический FK) |
| Booking | `booking_db` | `rentals` | `tenant_id` → `auth_db.users`, `property_id` → `property_catalog_db.properties` |
| Payment | `payment_db` | `transactions` | `rental_id` → `booking_db.rentals`, `tenant_id` → `auth_db.users` |
| Messaging | `messaging_db` | `messages` | `sender_id`, `receiver_id` → `auth_db.users`, `property_id` → `property_catalog_db.properties` |

## 4. Схема взаимодействия микросервисов

### 4.1. Архитектурная схема

```
                          ┌──────────────────────────────┐
                          │       API Gateway             │
                          │   (маршрутизация, rate-limit, │
                          │    агрегация ответов)         │
                          └──────┬───────┬───────┬───────┘
                                 │       │       │
                ┌────────────────┤       │       ├────────────────┐
                ▼                ▼       ▼       ▼                ▼
┌───────────────────┐ ┌──────────────┐ ┌──────────┐ ┌───────────────────┐ ┌──────────────┐
│    Auth Service   │ │   Property   │ │  Booking │ │  Payment Service  │ │  Messaging   │
│    (port 8081)    │ │   Catalog    │ │  Service │ │    (port 8084)    │ │   Service    │
│                   │ │ (port 8082)  │ │ (p.8083) │ │                   │ │  (port 8085) │
│    ┌───────────┐  │ │              │ │           │ │    ┌───────────┐  │ │              │
│    │  auth_db  │  │ │ ┌──────────┐ │ │┌─────────┐│ │    │ payment_db│  │ │┌────────────┐│
│    └───────────┘  │ │ │property_ │ │ ││booking_ ││ │    └───────────┘  │ ││messaging_db││
│                   │ │ │catalog_db│ │ ││   db    ││ │                   │ │└────────────┘│
└────────┬──────────┘ │ └──────────┘ │ │└─────────┘│ └────────┬──────────┘ └──────┬───────┘
         │            └──────┬───────┘ └─────┬──────┘          │                   │
         │                   │               │                 │                   │
         └───────────────────┼───────────────┼─────────────────┼───────────────────┘
                             │               │                 │
                    ┌────────▼───────────────▼─────────────────▼──────────┐
                    │                 Message Broker                       │
                    │              (RabbitMQ / Apache Kafka)              │
                    │                                                     │
                    │  Топики:                                            │
                    │  · user.registered  · user.updated                  │
                    │  · property.created · property.archived             │
                    │  · rental.created   · rental.status_changed         │
                    │  · payment.completed · payment.refunded             │
                    └─────────────────────────────────────────────────────┘
```

### 4.2. Способы взаимодействия

#### Синхронное (REST/gRPC)

Применяется для операций, требующих немедленного ответа:

| Источник | Приёмник | Операция | Причина REST |
|----------|----------|----------|--------------|
| API Gateway | Auth Service | Проверка JWT | Должен быть синхронным на каждом запросе |
| Booking Service | Property Catalog | Получение цены и статуса свойства | Нельзя создать бронь без подтверждения доступности |
| Booking Service | Auth Service | Проверка роли пользователя | Нужно немедленное решение |
| Property Catalog | Auth Service | Проверка существования владельца | Валидация при создании свойства |
| Messaging Service | Booking Service | Проверка участия в аренде | Нельзя отправить сообщение без проверки |

#### Асинхронное (Message Broker)

Применяется для событий, уведомлений и распределённых транзакций (Saga):

| Топик | Публикует | Подписчики | Назначение |
|-------|-----------|------------|------------|
| `user.registered` | Auth Service | Property Catalog, Booking, Messaging | Кэширование базовой информации о пользователе |
| `user.updated` | Auth Service | Property Catalog, Booking | Актуализация кэша пользовательских данных |
| `user.deleted` | Auth Service | Property Catalog, Booking, Messaging | Очистка кэша, проверка orphan-записей |
| `property.created` | Property Catalog | Booking | Актуализация доступных свойств |
| `property.price_changed` | Property Catalog | Booking | Актуализация цен в pending-бронированиях |
| `property.archived` | Property Catalog | Booking, Messaging | Блокировка новых бронирований |
| `rental.created` | Booking Service | Payment, Messaging | Уведомление о новом бронировании |
| `rental.status_changed` | Booking Service | Payment, Messaging | Синхронизация статуса, запуск возвратов |
| `payment.completed` | Payment Service | Booking | Saga: перевод rental.status → paid |
| `payment.failed` | Payment Service | Booking | Saga: обработка отказа платежа |
| `payment.refunded` | Payment Service | Booking | Saga: перевод rental.status → cancelled |

### 4.3. Паттерны отказоустойчивости

| Паттерн | Где применяется | Описание |
|---------|-----------------|----------|
| **Circuit Breaker** | Все синхронные вызовы | При N последовательных ошибок вызовы блокируются на T секунд, затем пробный запрос (half-open). Используется библиотека `gobreaker` или `hystrix-go`. |
| **Retry with Backoff** | Все синхронные вызовы | Экспоненциальная задержка между повторами (100ms → 200ms → 400ms, до 3 попыток). |
| **Timeout** | Все синхронные вызовы | Максимальное время ожидания ответа — 5 секунд. Превышение → fallback или ошибка 504. |
| **Dead Letter Queue** | Все асинхронные события | Сообщения, не обработанные после N попыток, попадают в DLQ для ручного разбора. |
| **Idempotency Key** | Payment Service → Booking Service | Ключ идемпотентности предотвращает двойное списание при повторной отправке события. |

## 5. OpenAPI спецификации межсервисных эндпоинтов

### 5.1. Auth Service — внутреннее API

#### `POST /internal/auth/validate`

Проверка JWT-токена. Используется API Gateway при каждом защищённом запросе.

```yaml
POST /internal/auth/validate
  Summary: Валидация JWT токена
  Request Body:
    {
      "token": "eyJhbGciOiJIUzI1NiIs..."
    }
  Responses:
    200:
      {
        "valid": true,
        "user": {
          "id": 1,
          "email": "user@example.com",
          "role": "tenant",
          "full_name": "Иван Иванов"
        }
      }
    401:
      {
        "valid": false,
        "error": "token_expired",
        "message": "Срок действия токена истёк"
      }
```

#### `GET /internal/users/{id}`

Получение данных пользователя по ID. Используется всеми сервисами.

```yaml
GET /internal/users/{id}
  Summary: Получение данных пользователя
  Parameters:
    - name: id
      in: path
      required: true
      schema: { type: integer }
  Responses:
    200:
      {
        "id": 1,
        "email": "user@example.com",
        "full_name": "Иван Иванов",
        "phone": "+79161234567",
        "role": "owner"
      }
    404:
      {
        "error": "user_not_found",
        "message": "Пользователь с ID 999 не найден"
      }
```

#### `POST /internal/users/batch`

Пакетное получение данных пользователей. Используется для обогащения списков (например, список свойств с именами владельцев).

```yaml
POST /internal/users/batch
  Summary: Пакетное получение пользователей
  Request Body:
    {
      "ids": [1, 2, 3]
    }
  Responses:
    200:
      {
        "users": [
          {"id": 1, "full_name": "Иван Иванов", "role": "owner", "email": "ivan@example.com"},
          {"id": 2, "full_name": "Петр Петров", "role": "tenant", "email": "petr@example.com"},
          {"id": 3, "full_name": null, "role": null, "email": null}
        ],
        "not_found": [3]
      }
    400:
      {
        "error": "invalid_request",
        "message": "Массив ids не должен быть пустым и не должен превышать 100 элементов"
      }
```

#### `GET /internal/users/{id}/role`

Проверка роли пользователя (облегчённый запрос).

```yaml
GET /internal/users/{id}/role
  Summary: Проверка роли пользователя
  Parameters:
    - name: id
      in: path
      required: true
      schema: { type: integer }
  Responses:
    200:
      {
        "user_id": 1,
        "role": "owner"
      }
    404:
      {
        "error": "user_not_found",
        "message": "Пользователь с ID 999 не найден"
      }
```

### 5.2. Property Catalog Service — внутреннее API

#### `GET /internal/properties/{id}`

Получение данных свойства. Используется Booking Service для получения цены и статуса, Messaging Service для валидации.

```yaml
GET /internal/properties/{id}
  Summary: Получение данных свойства
  Parameters:
    - name: id
      in: path
```

## 6. Обработка ошибок межсервисного взаимодействия

### 6.1. Стандартный формат ошибки

Все внутренние API возвращают ошибки в едином формате:

```json
{
  "error": "код_ошибки",
  "message": "Человекочитаемое описание",
  "details": {},
  "timestamp": "2026-05-17T10:05:00Z",
  "trace_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

### 6.2. Каталог ошибок

| HTTP-код | Код ошибки | Описание | Возвращает |
|----------|-----------|----------|------------|
| 400 | `invalid_request` | Некорректный формат запроса | Все сервисы |
| 400 | `invalid_dates` | Некорректные даты | Booking Service |
| 400 | `invalid_amount` | Некорректная сумма | Payment Service |
| 401 | `token_expired` | Истёкший токен | Auth Service |
| 401 | `token_invalid` | Невалидный токен | Auth Service |
| 403 | `forbidden` | Недостаточно прав | Все сервисы |
| 404 | `user_not_found` | Пользователь не найден | Auth Service |
| 404 | `property_not_found` | Свойство не найдено | Property Catalog |
| 404 | `rental_not_found` | Аренда не найдена | Booking Service |
| 404 | `transaction_not_found` | Транзакция не найдена | Payment Service |
| 409 | `invalid_transition` | Недопустимый переход статуса | Booking Service |
| 409 | `overlapping_rental` | Пересечение дат аренды | Booking Service |
| 409 | `duplicate` | Дубликат (идемпотентность) | Payment Service |
| 422 | `already_processed` | Операция уже выполнена | Booking, Payment |
| 500 | `internal_error` | Внутренняя ошибка сервиса | Все сервисы |
| 502 | `upstream_error` | Ошибка при вызове зависимого сервиса | API Gateway |
| 503 | `service_unavailable` | Сервис временно недоступен | Все сервисы |
| 504 | `upstream_timeout` | Таймаут вызова зависимого сервиса | Все сервисы |

### 6.3. Стратегии обработки сбоев

| Сценарий | Стратегия |
|----------|-----------|
| Auth Service недоступен | API Gateway возвращает 503 для защищённых эндпоинтов; публичные продолжают работать |
| Property Catalog недоступен | Booking Service отклоняет создание бронирования (невозможно проверить цену); выдача списков возвращает закэшированные данные |
| Booking Service недоступен | Payment Service не принимает новые платежи; очереди сообщений буферизируют события |
| Payment Service недоступен | Платежи становятся в очередь; status="pending" до получения подтверждения; пользователь видит статус "платёж в обработке" |
| Message Broker недоступен | Сервисы продолжают работу синхронно; события накапливаются в outbox-таблицах и отправляются при восстановлении |

### 6.4. Паттерн Transactional Outbox

Для гарантированной доставки событий каждый сервис реализует паттерн Transactional Outbox:

```sql
-- Пример outbox-таблицы в Booking Service
CREATE TABLE outbox_events (
    id          BIGSERIAL PRIMARY KEY,
    aggregate_type VARCHAR(100) NOT NULL,   -- 'rental'
    aggregate_id   INTEGER NOT NULL,
    event_type     VARCHAR(100) NOT NULL,   -- 'rental.created', 'rental.status_changed'
    payload        JSONB NOT NULL,
    created_at     TIMESTAMP DEFAULT NOW(),
    published_at   TIMESTAMP,
    retry_count    INTEGER DEFAULT 0,
    status         VARCHAR(20) DEFAULT 'pending'  -- pending, published, failed
);
```

Outbox-процессор периодически (раз в 1 секунду) читает записи со статусом `pending` и публикует их в Message Broker. После успешной публикации статус меняется на `published`.

## 7. Распределённые транзакции и Saga-паттерн

### 7.1. Оплата бронирования (Saga: Reservation → Payment)

В монолите это была одна ACID-транзакция (создание Transaction + обновление Rental). В микросервисах применяется **Orchestrated Saga** с Booking Service в роли оркестратора.

```
Booking Service (оркестратор)           Payment Service           Message Broker
─────────────────────────────────────────────────────────────────────────────────
1. POST /rentals/:id/pay
   Проверка: rental.status == confirmed
   Проверка: tenant_id == user_id
   
2. [Saga: шаг 1]
   PATCH rental.status → "payment_pending"    ──→ rental.status_changed
   (промежуточный статус)
   
3.                                         POST /internal/payments
                                            {rental_id, tenant_id, amount,
                                             payment_method, idempotency_key}
                                            
4.                                         ← 201 Created
                                            (transaction created, status=success)
                                            
5.                                         ──→ payment.completed
                                            
6. [Saga: шаг 2]
   Получено событие payment.completed
   PATCH rental.status → "paid"
   
   Если payment.failed:
   PATCH rental.status → "confirmed"        ← откат шага 1

Диаграмма последовательности (успешный сценарий):

  Client      Booking Svc      Payment Svc      Message Broker
    │              │                │                  │
    │  POST /pay   │                │                  │
    │─────────────>│                │                  │
    │              │                │                  │
    │              │─┐ валидация    │                  │
    │              │<┘ статуса      │                  │
    │              │                │                  │
    │              │── status=payment_pending ────────>│
    │              │                │                  │
    │              │  POST /internal/payments          │
    │              │───────────────>│                  │
    │              │                │─┐ обработка      │
    │              │                │<┘ платежа        │
    │              │  <── 201 ──────│                  │
    │              │                │                  │
    │              │                │── payment.completed ──>│
    │              │  <── событие ──────────────────────│
    │              │                │                  │
    │              │── status=paid ───────────────────>│
    │              │                │                  │
    │  <── 200 ────│                │                  │
    │  {rental,    │                │                  │
    │   transaction}                │                  │
```

### 7.2. Обработка сбоев Saga

| Точка сбоя | Действие оркестратора |
|------------|----------------------|
| Payment Service не ответил (timeout) | Повтор с экспоненциальной задержкой (3 попытки). Если все неудачны — rental.status возвращается в `confirmed`, платёж остаётся `pending` в payment_db. |
| Payment Service вернул ошибку (4xx) | Без повтора — rental.status возвращается в `confirmed`, ошибка возвращается клиенту. |
| Событие `payment.completed` потеряно | Payment Service имеет outbox; событие будет доставлено при восстановлении брокера. |
| Booking Service упал после шага 1, но до шага 6 | При восстановлении: сканирует rentals в статусе `payment_pending` дольше N минут, запрашивает статус платежа через `GET /internal/transactions/rental/{id}`, применяет соответствующее изменение статуса. |

### 7.3. Компенсирующие транзакции

| Операция | Компенсация |
|----------|-------------|
| `rental.status → payment_pending` | `rental.status → confirmed` (возврат к предыдущему статусу) |
| `transaction.status → success` | Создание refund-транзакции с `type: refund` |
| Изменение цены свойства | Не требует компенсации: цена фиксируется в `rental.total_price` на момент создания бронирования |

## Заключение

Предложенная микросервисная архитектура решает ключевые проблемы монолита:

1. **Независимое масштабирование** — catalog-сервис можно масштабировать под нагрузку поиска, не затрагивая остальные.
2. **Изоляция сбоев** — отказ messaging-сервиса не блокирует создание бронирований.
3. **Независимый деплой** — команды могут выпускать изменения независимо, в своём темпе.
4. **Database-per-service** — каждая БД оптимизирована под свой паттерн доступа (индексы, схема).
5. **Слабая связность** — сервисы зависят только от API-контрактов, а не от внутренней реализации друг друга.

**Ключевые риски:**
- Увеличение сложности отладки (распределённая трассировка обязательна)
- Дополнительная задержка на межсервисные вызовы (~5-20ms на вызов)
- Необходимость обработки частичных сбоев (Circuit Breaker, Retry, Saga)
- Усложнение CI/CD и мониторинга

Все эти риски адресованы применением описанных паттернов отказоустойчивости, централизованным мониторингом и поэтапным планом миграции.
